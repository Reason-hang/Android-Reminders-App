## 分层与变化点

- UI 只依赖 ViewModel；业务规则放在 UseCase；持久化通过 Repository 接口隔离。
- 内容、配置和逻辑分离；稳定维度保持简单，只有高概率变化点预留接口。
- 任何跨模块依赖必须保持单向，禁止 UI 直接操作 DAO 或 AlarmManager。

## 提醒事件规则

- `ADVANCE`、`DUE`、`SNOOZE` 使用独立 kind 和 requestCode。
- 只有 `DUE` 推进重复；Snooze 不改变重复模板或正式时间。
- 提前时间不得大于或等于重复周期。
- 操作至少携带 `reminderId + alarmId + kind + occurrenceTime`。
- 禁止无实例的 `stopService()`；旧事件不能误停新事件。

## 一致性与恢复

- 所有校验必须先于取消旧闹钟。
- Room 和 AlarmManager 不是同一事务；任一侧失败都要恢复、撤销或记录待重试状态。
- Receiver 推进重复时使用 occurrence 条件更新，防止重复广播二次推进。
- 开机、应用更新、精确闹钟权限变化和用户重新打开 App 时可重建闹钟。
- 单条恢复失败不能阻断其他提醒。

## Room 变更规则

字段变化必须同步检查 Model、Entity、Converter、Mapper、DAO、Repository、UseCase、ViewModel、UI、Migration、Schema 和测试。禁止 destructive migration；`alarmId` 必须有数据库唯一约束。

## 通知与隐私

- 前台服务保活通知与用户业务通知分离。
- 业务通知使用公开锁屏预览和 FullScreenIntent；最终展示仍受 Android/HyperOS 权限影响。
- 日志只记录内部标识和状态，不记录标题、备注正文。
- 不增加 `INTERNET`，不提交 keystore、密码、token 或本地配置密钥。

## 交付与证据

源码修改、单元测试、Lint、Release 构建、APK 验签、普通真机测试和目标真机测试是不同证据层，不能互相替代。APK 必须使用 `ReminderApp-v<version>.apk` 命名，并记录版本、大小、签名和 SHA-256。
