## 核实结论

报告列出的 N1-N4 在当前代码中均可复现或直接核实，根因判断正确。后续修订明确给出 N5、N6 的编号、位置和复现条件；两项均已复核为真实的防御性缺口并纳入 v1.6 修复。

## 已修复问题

| 编号 | 根因 | 修复 | 回归证据 |
|---|---|---|---|
| N1 | `WORKDAYS` 固定 `Asia/Shanghai`，其他规则使用设备时区 | 每次计算统一读取设备时区，并传给工作日和月度规则 | 非北京时间设备时区下，周五工作日正确推进到当地周一 |
| N2 | 标题/备注上限只在编辑 UI | 新增 `ReminderContentValidator`，新增和编辑 UseCase 均先校验 | 超长标题、超长备注被拒绝，且不写库、不替换闹钟 |
| N3 | 精确闹钟权限只随提醒列表 Flow 重算 | ViewModel 使用独立 StateFlow；列表首次进入和 `ON_RESUME` 时刷新 | 源码契约测试确认生命周期刷新入口 |
| N4 | 缺少可供 v3→v4 迁移测试使用的历史 Schema | 从仓库 v3 历史源码实际生成 `3.json`；新增 Room `MigrationTestHelper` 仪器测试 | Schema 契约测试通过；仪器测试源码已编译 |
| N5 | 脏 `alarmId` 用 `id` 取模派生负值；相差 `2,147,483,646` 的 ID 会碰撞，进而导致唯一索引创建失败 | 所有非正或重复值按 `id` 排序，以 `ROW_NUMBER()` 分配连续负值；不与保留的正值或历史负值碰撞 | JVM 契约测试覆盖 CTE；仪器测试构造重复、零、单独负值和 `Int.MAX_VALUE` 主键，已编译待设备执行 |
| N6 | 四个 Converter 直接 `enum.valueOf()`；损坏数据或未来漏迁移的枚举重命名会终止查询 | `status→PENDING`、`repeatType→NONE`、`advanceType→NONE`、`advanceUnit→HOURS`，且只捕获 `IllegalArgumentException` | 4 个非法值和 4 个全枚举正常映射 JVM 回归用例先红后绿；工程约束要求枚举改名必须配套 Migration |

## 未作为故障处理的建议

- `BootReceiver` 的恢复策略可在未来评估 WorkManager，但当前 `goAsync()`、异常隔离和 `finish()` 逻辑没有已复现缺陷。
- 长期停机后的重复计算可在未来做性能优化；现有提醒频率和个人使用规模下不是当前性能问题。
